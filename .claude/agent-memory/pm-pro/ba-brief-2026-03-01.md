# BA Brief for PM: Research-Backed Feature Roadmap for Step Next
## Date: 2026-03-01
## Status: READY FOR PM PROCESSING

## SOURCE DOCUMENTS ANALYZED
- **1.pdf** -- "Life progression visualization: the RPG skill tree for real life" -- Competitive landscape of 17+ gamified life-tracking apps, influencer frameworks (Athlean-X, Tate, Markarian, Goggins, Hamza Ahmed), 6 psychological mechanisms (SDT, Flow Theory, Information Gap, Zeigarnik, Goal Gradient, Endowed Progress), game design patterns (4 skill tree types), Duolingo path redesign case study, Yu-kai Chou's Octalysis Framework.
- **2.pdf** -- "The hidden path: progressive disclosure for life goals" -- 7 converging research traditions proving hidden future steps work (Bandura proximal goals, Sweller cognitive load, Masicampo/Baumeister open loops, Loewenstein info gap, Gollwitzer implementation intentions, Baumeister/Schwartz decision fatigue, Kivetz goal gradient), real-world implementations (exposure therapy, 12-step, military basic training, Bloom's mastery learning, game fog-of-war), ADHD community demand validation, design objections and solutions.

## KEY STRATEGIC FINDING
**No existing product combines "hidden future steps + multi-step life goals + progressive disclosure" as its core value proposition.** This is genuine whitespace in an $11.4B habit-tracking market (projected $43.9B by 2034). Step Next must own this position.

The linchpin design principle (Masicampo & Baumeister): "A plan that exists but isn't loaded into working memory provides cognitive relief without cognitive burden." The SYSTEM holds the future so the USER doesn't have to.

## CURRENT CODEBASE STATE
- React 19.2 + Vite 7.3, localStorage, no backend, plain CSS (BEM naming), UI in Ukrainian
- 4 step types: current (indigo/blue), free (green), milestone (gold), locked (grey with blur)
- Basic CRUD: add/delete spheres, add/delete/toggle steps. No drag-and-drop, no reorder.
- No animations, no progress indicators, no onboarding, no fog-of-war logic, no XP system.
- Key files: src/App.jsx, src/components/Sphere.jsx, src/components/StepNode.jsx, src/components/AddStepForm.jsx, src/components/AddSphereForm.jsx, src/utils/storage.js

## PRIORITIZED TASKS (ICE scored: Impact x Confidence x Ease, each 1-10)

### SPRINT 1: CORE UX FOUNDATIONS (Quick Wins, 1-2 days each)

**TASK 1: Implement 5 Visual Node States** | ICE: 9x9x7=567 | Size: M | Priority: P1-high
- **Problem**: Only 4 step types exist with no visual distinction between "available next" and "far future locked"
- **Research**: 1.pdf p.5-6 -- five distinct visual states form the foundation of any progression UI
- **Solution**: Refactor StepNode to support 5 visual states:
  1. **Completed** -- full color, checkmark overlay, subtle completed glow, tap to review
  2. **Active/Current** -- pulsating CSS animation, larger node (24px vs 20px), bright neon accent, primary interaction
  3. **Available (next)** -- visible but 70% opacity, tappable to preview and start
  4. **Locked** -- grey (#374151), lock icon, 50% opacity, tap shows prerequisites text
  5. **Hidden (fog of war)** -- silhouette shape, completely obscured content, non-interactive
- **Files to modify**: StepNode.jsx, StepNode.css
- **Acceptance criteria**: Each state visually distinct, smooth CSS transitions between states, proper cursor/pointer behavior per state
- **Labels**: feature, frontend, P1-high, M

**TASK 2: Celebration/Unlock Animations** | ICE: 8x8x8=512 | Size: M | Priority: P1-high
- **Problem**: Completing a step only toggles opacity/strikethrough. Zero emotional reward = zero dopamine = zero retention.
- **Research**: 1.pdf p.6 -- "Celebration animations serve as 'fiero' moments (emotional peaks of achievement) that punctuate the journey and encode positive memories"
- **Solution**: Three celebration tiers:
  1. **Step completion**: color pulse on node + brief particle burst (CSS keyframes)
  2. **Milestone reached**: confetti animation + "MILESTONE REACHED" banner (consider canvas-confetti ~3KB)
  3. **New step revealed**: smooth fade-in from fog with glow effect + "NEW STEP UNLOCKED" micro-toast
- **Files to modify**: StepNode.jsx, StepNode.css, Sphere.jsx (new: CelebrationOverlay component)
- **Acceptance criteria**: 3 distinct tiers work, animations < 2 seconds, no jank on mobile, toggle in settings to disable
- **Labels**: feature, frontend, P1-high, M

**TASK 3: Endowed Progress Effect** | ICE: 9x7x9=567 | Size: S | Priority: P1-high
- **Problem**: New spheres start with blank "current state" + locked "???". User feels zero momentum.
- **Research**: 1.pdf p.4 -- Nunes & Dreze (2006): starting at "Level 1" vs "Level 0" meaningfully increases completion rates. Pre-filled loyalty cards complete faster.
- **Solution**:
  1. New sphere creation wizard (2 steps): "What's your current state?" -> "What's one free action you can take?"
  2. First step auto-marked as starting point, sphere shows "Level 1" badge
  3. Progress indicator starts at ~15% (not 0%) to leverage endowed progress
- **Files to modify**: AddSphereForm.jsx, App.jsx, storage.js, Sphere.jsx, Sphere.css
- **Acceptance criteria**: Guided sphere creation flow, Level 1 badge visible, progress bar starts non-zero
- **Labels**: feature, frontend, P1-high, S

**TASK 4: Step Count Progress Indicator** | ICE: 8x9x7=504 | Size: S | Priority: P1-high
- **Problem**: No visibility of progress. User can't tell how far they've come or how far to go.
- **Research**: 1.pdf p.5 -- dual system (continuous + discrete), Bonezzi et al. (2011) on framing. 2.pdf p.6 -- hiding ALL future steps sacrifices goal gradient acceleration, so show "Step 3 of 7" counter.
- **Solution**:
  1. Sphere header shows "Krok 3 z 7" text indicator
  2. Thin XP-style progress bar below sphere name (continuous feedback)
  3. Dynamic framing: <50% progress shows "Completed X steps!", >50% shows "Only X steps left!"
- **Files to modify**: Sphere.jsx, Sphere.css
- **Acceptance criteria**: Counter renders correctly, progress bar animates on completion, framing switches at 50% threshold
- **Labels**: feature, frontend, P1-high, S

### SPRINT 2: CORE DIFFERENTIATOR (3-5 days each)

**TASK 5: Progressive Disclosure / Fog-of-War Engine** | ICE: 10x8x5=400 | Size: L | Priority: P0-critical
- **Problem**: THE core differentiator is missing. All steps visible to user. No progressive revelation.
- **Research**: 2.pdf (entire document). Seven research traditions converge: hidden steps reduce cognitive load, create curiosity pull, automate next actions, reduce decision fatigue, and create goal gradient acceleration cycles.
- **Solution**: View-layer fog computation (don't change data model):
  1. `computeVisualState(step, index, completedCount, totalCount)` utility function
  2. Rules: completed steps -> "completed" state; first incomplete -> "active"; second incomplete -> "available"; third incomplete -> "locked" (show title only); fourth+ -> "hidden" (silhouette)
  3. Step count indicator shows total hidden: "Krok 3 z 7 (4 prykhovano)"
  4. Fog state computed on render, not stored -- data model stays clean
- **Files to modify**: new src/utils/fogEngine.js, Sphere.jsx (wrap steps through fog computation), StepNode.jsx (consume visual state)
- **Acceptance criteria**: Auto-reveal on completion, hidden steps show as silhouettes with count, progression persists in localStorage, no data model changes
- **Dependencies**: Requires TASK 1 (5 visual states)
- **Labels**: feature, frontend, P0-critical, L

**TASK 6: "Peek Ahead" Opt-in Feature** | ICE: 7x7x6=294 | Size: M | Priority: P2-medium
- **Problem**: SDT (Self-Determination Theory) emphasizes autonomy. Some users will feel controlled by hidden info.
- **Research**: 2.pdf p.5 -- autonomy concern. Solution: opt-in "peek ahead" that shows the plan exists without forcing processing.
- **Solution**:
  1. Eye icon toggle on sphere header (closed eye = fog active, open eye = all visible)
  2. When toggled on, hidden/locked nodes show in muted state (not full clarity) with "preview mode" indicator
  3. Per-sphere setting, stored in localStorage
  4. Default: fog ON (progressive disclosure active)
- **Files to modify**: Sphere.jsx, Sphere.css, storage.js (add peekAhead flag to sphere model)
- **Acceptance criteria**: Toggle works per sphere, visual distinction in peek mode, setting persists
- **Dependencies**: Requires TASK 5 (fog engine)
- **Labels**: feature, frontend, P2-medium, M

**TASK 7: Onboarding Flow** | ICE: 7x8x5=280 | Size: M | Priority: P2-medium
- **Problem**: No onboarding. User lands on demo data with no explanation of the product's unique value.
- **Research**: 2.pdf p.4-5 -- ADHD community wants simplicity; 1.pdf p.6 -- "aha moment" must demonstrate the fog-of-war mechanic immediately.
- **Solution**: 3-step interactive onboarding (not a tutorial -- user does real actions):
  1. "Welcome" -- explain concept in 1 sentence: "Your future reveals itself one step at a time"
  2. "Create your first sphere" -- guided sphere creation
  3. "Complete your first step" -- user sees fog-of-war in action, new step appears
  4. Store `onboarding_completed: true` in localStorage
  5. Richer demo data: 3 spheres (Sleep/Fitness/Finance) with 4-6 steps each showing all 5 states
- **Files to modify**: new src/components/Onboarding.jsx, App.jsx, storage.js
- **Acceptance criteria**: First-time user sees onboarding, "aha" < 60 seconds, skip option, demo data comprehensive
- **Labels**: feature, frontend, P2-medium, M

### SPRINT 3: STRATEGIC (This Quarter)

**TASK 8: Dual XP + Level System** | ICE: 7x8x4=224 | Size: L | Priority: P2-medium
- **Problem**: No gamification progression beyond step completion. No global sense of advancement.
- **Research**: 1.pdf p.5 -- dual system: continuous XP bar (micro-feedback) + discrete level-ups (celebration events). Levels are "far more socially communicable" ('I'm Level 12' vs 'I have 47,832 XP').
- **Solution**:
  1. Per-sphere XP: each step completion = XP based on step type (free=10, milestone=50, etc.)
  2. Global "Life Level" aggregating all spheres
  3. Level thresholds with celebration (reuse TASK 2 infrastructure)
- **Labels**: feature, frontend, P2-medium, L

**TASK 9: Domain Template Library** | ICE: 6x7x4=168 | Size: L | Priority: P2-medium
- **Problem**: Blank spheres require too much creative effort to populate. Cold-start problem.
- **Research**: 1.pdf p.1 -- Skilltree has pre-built paths. 2.pdf p.1 -- Fabulous introduces habits via "Journeys."
- **Solution**: 5-10 pre-built templates (Sleep, Morning Routine, Fitness Beginner, Financial Foundation, Social Skills, Reading Habit), fog-of-war applied by default
- **Labels**: feature, frontend, P2-medium, L

**TASK 10: Information Gap Design for Locked/Hidden Nodes** | ICE: 7x6x4=168 | Size: M | Priority: P2-medium
- **Problem**: Hidden nodes show nothing. Zero curiosity = zero pull.
- **Research**: 1.pdf p.4 -- Loewenstein's inverted U-curve: "knowing nothing produces no curiosity, knowing everything closes the gap." Optimal: reveal just enough (title silhouette, category tag, vague hint).
- **Solution**: Three-tier info density decreasing with distance from current step. Near-locked shows category + hint; far-locked shows silhouette only; hidden shows blank shape.
- **Labels**: feature, frontend, P2-medium, M

**TASK 11: Monetization Architecture Prep** | ICE: 6x6x3=108 | Size: M | Priority: P3-low
- **Problem**: No infrastructure for future monetization.
- **Research**: Market is $11.4B. Competitors: subscriptions (Habitica), one-time (LifeUp $5-8), freemium (Fabulous), community ($49/mo Tate).
- **Solution**: Data model support for premium flags, user tiers, analytics event hooks. No actual paywall yet.
- **Labels**: infrastructure, P3-low, M

**TASK 12: Data Export/Import** | ICE: 6x8x6=288 | Size: S | Priority: P2-medium
- **Problem**: localStorage = data loss risk. Users need safety net.
- **Solution**: JSON export/import buttons in settings. Critical for trust before asking for payment.
- **Labels**: feature, frontend, P2-medium, S

1. **ADHD/neurodivergent** -- task paralysis clinically documented, strongest product-market fit
2. **Minimalist productivity** -- "intentional limitation" valued, HN/Reddit demand signals
3. **Young men in self-improvement** -- influencer landscape validates demand for "level up" tools
4. **Gamification enthusiasts** -- overlap with RPG/anime communities

## KEY DESIGN PRINCIPLES FROM RESEARCH
1. Use **hybrid skill tree model**: vertical primary axis + branching at milestones
2. Prioritize **White Hat gamification** (Development, Accomplishment, Curiosity) over **Black Hat** (Scarcity, Urgency)
3. Frame as **reward** ("You've unlocked your next step") NOT **deprivation** ("Your next step is locked")
4. Target **non-punitive** model (like Finch, not Habitica): no damage for missed days
5. **Progressive difficulty**: early steps take minutes, later steps take days/weeks
